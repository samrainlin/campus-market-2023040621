**AI 与我的分工关系：**
- **AI 擅长**：快速生成模板代码、解释技术概念、检查文档语法

- **关键原则**：AI 输出的代码一定要经过我的理解和验证后才能使用，不能直接复制粘贴

**AI 协作中的典型修正场景：**

1. **AI 生成 JWT 登录逻辑** — 项目仅用 JSON Server，没必要引入 JWT，手动删除
2. **AI 建议路由守卫检查登录** — 项目简单，只需在特定页面做条件判断，不引入全局路由守卫
3. **AI 把 Element Plus 写到技术栈** — package.json 中虽然安装了，但实际代码没用到，从 README 移除
4. **AI 建议对密码做前端加密** — Mock 项目中明文存储足够，不增加不必要的复杂度
5. **AI 生成的搜索组件包含后端接口调用** — 项目使用前端过滤，改为纯组件

## 8. 遇到的问题与解决方法

**问题 1：端口 3001 被占用，JSON Server 无法启动**

解决方法：用 PowerShell 查看占用端口的进程 ID，然后杀掉该进程。之后每次启动前都会先检查端口是否被占用。

**问题 2：刷新页面后，Pinia 中的用户状态丢失**

解决方法：在登录成功时用 localStorage 保存用户信息（`localStorage.setItem`），在 App.vue 的 onMounted 中读取 localStorage 恢复用户状态（`localStorage.getItem`），退出登录时清除。这样刷新后状态不丢失。

**问题 3：发布页 publisher 字段需要取当前用户**

解决方法：在 Pinia 的 user store 中定义 getter `displayName`，在发布页提交时读取 `userStore.displayName` 填入 publisher 字段。

**问题 4：AI 建议使用 JWT 和路由守卫，但项目仅用 JSON Server**

解决方法：手动删除 AI 生成的 JWT 和路由守卫代码。项目使用简单的账号密码校验 + localStorage 状态持久化，足够支撑前端实训目标。

**问题 5：搜索功能，keyword 变化时列表没有重新渲染**

解决方法：确保在 computed 中正确引用 keyword.value，并且列表数据依赖正确。Vue 3 的 computed 会自动追踪依赖，修复后搜索功能正常工作。

**问题 6：技术栈描述不准确（Day7 审查发现）**

解决方法：检查实际代码是否使用了 Element Plus 组件，确认项目仅使用原生 HTML + CSS，没有引入任何 UI 组件库。从 README 的技术栈部分移除 Element Plus。

## 10. 项目不足与改进方向

> 综合验收后需要诚实地记录项目当前的不足，以及接下来可以优化的方向。这也是 AI 复盘中很重要的一环——AI 生成的内容往往偏向"完美陈述"，需要人工补上真实的不足。

### 10.1 功能层面的不足

**未实现真正的后端接口：** 当前全部数据依赖 JSON Server 的 db.json，没有真正的后端服务。如果要迁移到真实后端（Node.js + Express 或 Spring Boot），需要：
- 设计 RESTful API 规范，每个接口都有明确的请求/响应结构
- 将 users 数据迁移到真实数据库（MySQL/MongoDB），密码改为哈希存储
- 添加 CORS、鉴权、防 CSRF、防 XSS 等安全措施

**未实现消息通知动态逻辑：** MessageView.vue 目前只是静态展示页面，没有实现通知列表、未读标记、点击跳转等功能。改进方向：
- 在 db.json 中新增 messages 数据集合（字段：id、title、content、time、read）
- 在 userStore 中添加未读消息数量的状态
- 在 AppHeader 导航栏展示未读消息角标

**未实现真实的详情页：** 当前所有业务信息都是在卡片中展示简要信息，没有详情页（/trade/:id、/lost-found/:id 等）。改进方向：
- 在 router/index.ts 中添加带参数的详情路由
- 新增 TradeDetailView.vue 等页面，onMounted 时通过 `route.params.id` 请求单条数据
- 在列表页卡片上添加"查看详情"路由跳转

**用户中心"我的发布"逻辑不够完整：** 目前只实现了从 Pinia 读取 publisher 匹配列表，没有实现"删除/编辑我发布的信息"功能。改进方向：
- 在 UserCenterView.vue 中为每条"我的发布"添加删除按钮
- 删除时调用对应 API 的 delete 方法，同步更新列表

**收藏功能未持久化：** 收藏状态目前只存在 Pinia 内存中，刷新页面后收藏状态会丢失。改进方向：
- 在 userStore 中新增 favoriteIds 字段，登录时从 users 数据读取并恢复
- 或者使用 localStorage 单独保存 favorites（类似用户状态）

### 10.2 代码与架构层面的不足

**未引入统一的错误处理：** 目前每个 API 调用都在页面组件里写 try/catch，错误处理逻辑分散。改进方向：
- 在 src/api/http.ts 中添加 Axios 响应拦截器，统一处理网络错误和服务端错误
- 封装一个全局的 `showError(message)` 方法，替换各页面的零散 alert

**TypeScript 类型定义还可以更严谨：** 部分页面中使用了 `any` 类型（LoginView.vue、RegisterView.vue 中有少量历史遗留）。改进方向：
- 统一使用 interface 或 type 定义所有数据结构
- 开启 tsconfig 中 `noImplicitAny: true` 严格检查

**样式文件组织不够规范：** 目前主要依赖 src/assets/main.css 一个文件维护全局样式，各页面的 scoped style 中也有重复的类名。改进方向：
- 引入 CSS 变量统一管理主题色（蓝色主色调、字体、间距）
- 将通用样式抽离为多个 CSS 文件（base.css / layout.css / components.css）
- 统一所有卡片组件的圆角、阴影、间距

**缺少单元测试：** 项目目前只有手动功能走查，没有任何自动化测试。改进方向：
- 引入 Vitest 写简单的单元测试（userStore 的 login/logout 逻辑、表单校验函数）
- 引入 Vue Test Utils 写组件测试（SearchBar 的输入和清空、FormField 的错误展示）

### 10.3 交互体验层面的不足

**表单校验反馈不够友好：** 目前表单错误提示使用 alert 弹窗，用户体验较差。改进方向：
- 引入行内错误提示（红色文字显示在输入框下方）
- 添加失焦校验（blur 时即时校验该字段，不需要等提交才报错）
- 添加输入框边框变红等视觉反馈

**列表页没有分页功能：** 目前所有列表页都是一次性加载全部数据，数据量大时页面会很长。改进方向：
- 在 db.json 中增加更多测试数据（50+ 条）
- 在各列表页添加原生分页组件（每页 10 条，上一页/下一页/页码）
- 或使用虚拟滚动只渲染可视区域

**搜索功能只支持二手交易页：** 目前只有 TradeView.vue 完整实现了搜索，其他列表页（失物招领、拼单搭子、跑腿委托）虽然有搜索组件展示，但过滤逻辑不一致。改进方向：
- 统一所有列表页的搜索过滤逻辑
- 搜索关键词高亮（在标题和描述中匹配的部分用黄色背景标记）

## 11. 今日反思

**关于项目开发：**

经过 7 天的实训，我完成了一个能正常运行的前端小项目。从最开始的项目结构梳理，到逐步添加页面、数据、功能、状态管理、交互优化，最后完成文档和检测，这个过程让我对 Vue 3 工程化有了系统的理解。

最深的感受是：前端项目的开发需要层层递进。先有页面和路由，再有数据和列表，然后有表单和状态管理，最后有交互优化和文档。如果跳过某一步直接上复杂功能，代码会变得很难维护。

**关于综合验收与项目展示：**

Day7 的综合验收过程让我认识到，"做完功能"和"可以展示"是两个完全不同的阶段。功能做完只是第一步，要真正能完成项目展示，还需要：
- 确保构建通过 `npm run build` 无报错
- 确保检测脚本通过
- 准备清晰的演示流程和说明文档
- 准备项目展示的提纲和答辩要点
- 检查证据卡是否齐全、内容是否真实

在准备综合验收时我走查了 11 个核心步骤（首页 → 列表 → 搜索 → 注册 → 登录 → 发布 → 收藏 → 个人中心 → 退出 → 未登录拦截 → 错误状态 → 构建），每一步都验证通过后才确认项目可以展示。

**关于状态管理：**

Pinia 让我理解了什么是「跨组件的共享状态」。用户登录后，导航栏要显示用户名、发布页要取用户名称作为发布者、个人中心要展示用户资料 —— 这些分散在不同组件中的信息如果都用 prop 传递会非常繁琐，而 Pinia 提供了一个集中的 store，让所有组件都能访问同一个状态。

localStorage 的配合使用也让我明白了「前端状态持久化」的基本思路。Pinia 的状态只在内存中，浏览器刷新就会丢失，而 localStorage 提供了一个简单的方式来保存和恢复状态。

**关于 Mock 数据：**

JSON Server 让我能在没有后端的情况下完成整个前端开发。这让我理解了前端在真实项目中的位置：前端负责 UI 和交互，数据由后端提供。Mock 数据让我能独立开发、独立测试、独立演示功能。

**关于 AI 复盘：**

7 天的开发中 AI 一直是我的助手。它帮我：快速写出组件模板、解释技术概念、生成代码结构建议。但 AI 复盘时我也发现 AI 有几个典型问题：

1. **AI 偏好复杂方案** —— 它建议的 JWT、路由守卫等，对这个简单项目是过度设计
2. **AI 的描述可能不准确** —— 它把 Element Plus 写进技术栈，但实际上项目并未使用（我在 Day7 综合验收时发现并修正了）
3. **AI 不了解项目实际情况** —— 它建议的搜索后端接口调用，和项目的前端过滤方案不符
4. **AI 生成的"不足与改进方向"总是泛泛而谈** —— AI 更擅长说"可以优化"，但很难说出"具体在我项目里哪一行、哪个组件有问题"

所以我的总结是：AI 是很好的助手，但最终判断必须由我自己做。代码是否合理、功能是否必要、文档是否准确，都需要我理解后再做决定。AI 复盘时尤其要人工审查——AI 不会告诉你它生成的内容有什么遗漏，需要我自己对照项目现状逐条检查。

**关于项目验收：**

Day7 的检测脚本（49 项检测）让我意识到一个好的项目交付不只是功能做完，还需要代码规范、文档完整、构建通过。`npm run build` 能通过说明类型检查没有问题，检测脚本能通过说明目录结构和文件命名符合预期。

**最终收获：**

通过这 7 天，我理解了一个 Vue 3 前端项目从 0 到 1 的完整开发路径，掌握了 Vue Router、Pinia、Axios、JSON Server 的基本使用方法，并体验了 AI 辅助开发的正确姿势。虽然这只是实训项目，但我对未来做更大规模的真实项目有了更多信心。

在综合验收和 AI 复盘的最后，我最大的收获是：**代码能跑只是开始，能解释清楚、能演示顺畅、能诚实地面对不足，才是真正完成了一个项目。**
