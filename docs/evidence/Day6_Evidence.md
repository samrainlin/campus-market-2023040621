## 2. 注册登录设计说明

注册流程大概是这样的：

1. 用户在注册页填用户名、密码、确认密码、姓名、学院、年级
2. 表单先自己检查一下：用户名不少于 3 位，密码不少于 6 位，两次密码要一样
3. 调 getUsers() 先查一下 users 里有没有重名的用户
4. 用户名重复就提示错误，不重复就调 createUser() 把用户写到 db.json
5. 注册成功后跳到登录页

登录流程是这样的：

1. 用户在登录页输入用户名和密码
2. 调 getUsers() 拿到所有用户
3. 在前端遍历找有没有 username 和 password 都匹配的
4. 找到匹配的就把这个用户存到 Pinia，同时存到 localStorage
5. 登录成功后跳到个人中心

刷新页面后登录状态能保留，是因为在 App.vue 里的 onMounted 调了 restoreLogin()，从 localStorage 里把用户信息读出来，再塞回 Pinia。

退出登录就反过来，点导航栏的退出按钮，调 userStore.logout()，清掉 Pinia 和 localStorage 里的东西，跳回登录页。

## 4. 状态持久化说明

Day5 做完后我发现刷新页面后登录状态就没了，后来才想明白：Pinia 的状态只存在 JS 内存里，页面一刷新内存就重置了，所以登录状态自然就没了。

为了解决这个问题，Day6 加了 localStorage 来存用户信息。

用 localStorage 的原因：

- 它能长期保存，不会因为刷新页面或关浏览器就丢了
- API 很简单，就 setItem、getItem、removeItem 三个
- localStorage 只能存字符串，正好可以用 JSON.stringify 把用户对象转成字符串存进去
- 对学生项目来说够简单，不用额外装别的库

具体怎么用的：

- 登录成功时：localStorage.setItem('campus-market-current-user', JSON.stringify(user))
- 页面刚加载时：在 App.vue 的 onMounted 里调 restoreLogin() 从 localStorage 读出来恢复
- 退出登录时：localStorage.removeItem('campus-market-current-user')

这里要说明一下：这种做法只是学生项目用来练手的模拟登录，不是生产环境安全认证。真实项目里密码不会明文存，也不会在前端校验密码。

## 8. 功能边界说明

要特别说明一下：本项目的注册登录功能只是前端实训用的模拟功能，**不是生产环境安全认证系统**。

具体有这些边界：

- 密码没有加密，db.json 里 users 的密码是明文存的，真实项目中必须用 bcrypt 之类的哈希算法加密
- 没有 JWT，也没有 Token 过期机制
- 没有权限路由，除了发布和个人中心的部分交互外，其他页面不区分登录未登录
- 没有短信验证码或邮箱验证，注册流程只有前端表单校验
- 没有找回密码功能
- 所有校验逻辑都在前端完成，用户可以改 localStorage 或直接请求 API 绕过检查
- localStorage 只是客户端存储方案，数据存在用户本机，不适合存真实用户数据

理解这些边界很重要：Day6 学的是前端状态怎么管理和共享，不是怎么做安全登录系统。真实生产环境的登录需要后端、HTTPS、密码加密、Token 过期、防 CSRF/XSS 这些保障，不是前端一个人能搞定的。

## 10. 遇到的问题与解决方法

问题1：注册时提示「注册失败，请检查网络或 Mock 服务是否正常运行」

一开始我以为是代码写错了，反复看了 RegisterView.vue 的表单和提交逻辑，没看出什么错。后来看终端才发现 npm run mock 启动时报了 "EADDRINUSE: address already in use"，3001 端口被别的进程占了。

解决方法：用 PowerShell 跑 Get-NetTCPConnection -LocalPort 3001 看哪个进程占了端口，按 PID 找到那个进程 Stop-Process -Id [PID] -Force 杀掉，然后重新跑 npm run mock，注册功能就正常了。

问题2：登录后刷新页面，登录状态丢了

Day5 做完就有这个问题，但当时没解决。Day6 才想明白：Pinia 的状态只存在 JavaScript 内存里，页面一刷新内存就清了，状态自然就没了。

解决方法：登录成功时同时把用户信息存到 localStorage，并在 App.vue 的 onMounted 中调 restoreLogin() 从 localStorage 读出来恢复状态。退出登录时同时清掉 Pinia 和 localStorage。

问题3：登录页的错误信息不够清楚

一开始不管什么错都显示「账号或密码错误」，Mock 服务没启动也是这个提示，我查了好久才发现是服务没启动。

解决方法：改了 userStore 的 login() 方法，把网络请求放 try/catch 包起来，网络错误时抛「无法连接到 Mock 服务，请检查 JSON Server 是否启动」的错误；账号密码不匹配时抛「账号或密码错误」。登录页面读 error.message 动态显示，这样就能清楚知道是服务问题还是账号问题。

问题4：AI 生成的代码里有 JWT 相关逻辑

刚开始让 AI 写登录代码，它直接写了一整套 JWT 认证流程，包括 token 生成、token 解码、请求拦截器注入 token、路由守卫检查 token 有没有效果等，对 Day6 来说太复杂了，也和实训目标不搭。

解决方法：手动删了所有 JWT 相关代码，只留最简单的账号密码校验逻辑。具体做法：GET /users 拿全部用户，在前端数组里找匹配的；匹配成功就直接把用户对象存到 Pinia。这样代码更短，也更符合实训项目的学习目标。

问题5：搜索功能不起作用

二手交易页实现搜索时，一开始在 SearchBar 组件里把搜索词 emit 出来，但父组件的过滤逻辑位置放错了地方，导致 keyword 变化时列表没有重新渲染。

解决方法：检查 computed 的依赖对不对，确保 filteredList 的计算属性里用了 keyword.value。只要 keyword 有变化，computed 就会重新计算并自动更新模板。修完后搜索功能正常了。

## 11. 今日反思

今天做完 Day6，最大的收获是理解了「用户状态」到底是什么东西。之前以为就是个变量存一下就完了，实际上是贯穿整个应用的共享信息。通过 Pinia 管登录状态，看到了导航栏、发布页、个人中心这些页面怎么同时跟着一个状态的变化，对状态管理有了更具体的认识。

关于状态持久化，之前总以为这是后端或框架自动处理的事，今天亲手实现了 localStorage 的存和读的逻辑，才知道前端状态要开发人员主动保存和恢复。这种「内存中的状态会丢」的理解很关键，真实项目中这个问题会被更复杂的方案解决，但 Day6 的 localStorage 版本让我理解了最基本的原理。

关于功能边界，今天的实训让我意识到，学生项目里写的「登录/注册」和真实网站的登录不是一回事。真实系统需要后端服务、密码加密、HTTPS、Token、权限校验这些保障，而我们只是在前端模拟了一个流程。知道这个边界很重要，不会让我错误地以为当前写的代码可以直接上线用。

交互优化方面，通过加 loading、error、empty 状态，我意识到一个好的 Web 应用要考虑所有可能的状态，而不只是「成功」这一种状态。用户体验的好坏往往就体现在异常状态时反馈及不及时和友不友好，而不在于功能能不能用。

最后关于 AI 协作，也有新的体会：AI 会偏向给出行业里常用的或者完整的复杂方案，但作为学生，有时候需要的是最简化的、能帮我理解核心概念的实现。人工审查和筛选就变得很重要，不能直接拿 AI 生成的所有代码，而是要根据当前的学习阶段和项目需求做裁剪和调整。
