## 2. 注册登录设计说明

Day6 的注册/登录流程基于 JSON Server、Pinia 和 localStorage 实现，具体流程如下：

**注册流程：**
1. 用户填写用户名、密码、确认密码、姓名、学院、年级
2. 前端校验必填项和格式（用户名≥3字符，密码≥6字符，两次密码一致）
3. GET /users 查询现有用户列表，检查用户名是否重复
4. 若重复，显示错误提示；否则 POST /users 将新用户写入 db.json
5. 注册成功后跳转到登录页

**登录流程：**
1. 用户输入用户名和密码
2. GET /users 获取用户列表
3. 在前端遍历查找 username 和 password 都匹配的记录
4. 若匹配成功，将用户信息保存到 Pinia（userStore.currentUser 和 isLoggedIn=true）
5. 同时将用户信息 JSON 化后存入 localStorage，key 为 "campus-market-current-user"
6. 登录成功后跳转到个人中心页面

**状态恢复流程：**
1. App.vue 挂载时调用 userStore.restoreLogin()
2. 从 localStorage 读取 "campus-market-current-user"
3. 若存在有效数据，恢复到 Pinia 状态，保持用户登录状态
4. 刷新页面后登录状态不会丢失

**退出登录流程：**
1. 点击导航栏的「退出」按钮
2. 调用 userStore.logout()，清空 Pinia 中的 currentUser 和 isLoggedIn
3. 同时删除 localStorage 中的用户数据
4. 跳转到登录页面

## 4. 状态持久化说明

Day5 完成时我发现一个问题：刷新页面后，Pinia 中的用户状态会丢失。这是因为 Pinia 的状态仅存在于前端组件的内存中，页面刷新时 JavaScript 运行环境重新初始化，所有内存变量都被清理。

为了解决这个问题，Day6 引入了 localStorage 作为状态持久化的手段。

**为什么选择 localStorage：**

1. **持久性**：localStorage 存储的数据不会随页面刷新或关闭浏览器而丢失，可以长期保存
2. **简单易用**：只需 setItem/getItem/removeItem 三个 API 就能实现基本存取
3. **字符串存储**：localStorage 只支持字符串存储，正好可以用 JSON.stringify 保存用户对象
4. **适合实训场景**：不需要引入额外库（如 pinia-plugin-persistedstate），逻辑更简单，学生更容易理解

**具体实现方式：**

- 登录成功时：localStorage.setItem('campus-market-current-user', JSON.stringify(user))
- 页面加载时：在 App.vue 的 onMounted 中调用 userStore.restoreLogin()，从 localStorage 读取并解析 JSON
- 退出登录时：localStorage.removeItem('campus-market-current-user')

需要特别说明的是，这种方式只是前端实训项目的 Mock 方案，不是生产环境安全认证。真实项目中密码会在后端加密存储、Token 会有过期时间、敏感信息不应直接存入 localStorage。

## 8. 功能边界说明

**重要声明**：本项目的注册/登录功能基于 JSON Server、Pinia 和 localStorage 实现，仅用于前端实训和状态管理模拟，**不是生产环境安全认证系统**。

具体的功能边界如下：

1. **密码未加密**：db.json 中的 users 数据以明文形式存储密码，真实项目中必须使用 bcrypt 等哈希算法加密，并且不应在前端进行密码校验
2. **无 JWT**：本项目不使用 JSON Web Token，也没有实现基于 Token 的认证机制
3. **无权限路由**：所有页面（除个人中心和发布页的部分交互外）不区分登录用户和未登录用户的访问权限
4. **无短信验证码/邮箱验证**：注册流程仅使用前端表单校验，不发送验证消息
5. **无找回密码**：未实现忘记密码、重置密码等功能
6. **无后端安全校验**：所有校验逻辑在前端完成，用户可以通过修改 localStorage 或直接请求 API 绕过检查
7. **状态持久化仅前端**：使用 localStorage 保存用户信息是简单的客户端存储方案，刷新或关闭浏览器后数据存在本机，不适合真实用户数据

理解这些边界对学习很重要：Day6 学习的是「前端状态如何管理和共享」，而不是「如何实现一个安全的认证系统」。真实生产环境的登录系统需要后端参与、HTTPS 传输、密码加密存储、Token 过期机制、防 CSRF/XSS 等多方面保障。

## 10. 遇到的问题与解决方法

**问题1：注册失败，显示「请检查网络或 Mock 服务是否正常运行」**

一开始我以为是代码写得有问题，反复检查 RegisterView.vue 中的表单和提交逻辑，没有发现明显错误。后来在终端中发现 `npm run mock` 的 JSON Server 启动时出现了 "EADDRINUSE: address already in use" 报错，说明 3001 端口被其他进程占用了。

**解决方法**：用 PowerShell 执行 `Get-NetTCPConnection -LocalPort 3001` 查看占用进程，通过 PID 找到占用端口的进程并杀掉（`Stop-Process -Id [PID] -Force`），然后重新启动 Mock 服务，注册功能就正常了。

**问题2：登录后刷新页面，登录状态丢失**

Day5 完成时就发现了这个问题，但当时没有好的解决方案。Day6 时才明白：Pinia 的状态只存在于 JavaScript 内存中，页面刷新时内存被清理，状态自然就没了。

**解决方法**：在登录成功时同时将用户信息存入 localStorage，并在 App.vue 的 onMounted 生命周期中调用 restoreLogin() 读取 localStorage 恢复状态。退出登录时同时清除 Pinia 和 localStorage 中的数据。

**问题3：登录页的错误信息不够明确**

一开始无论什么错误都显示「账号或密码错误」，包括 Mock 服务没有启动时也是这个提示，导致我排查了很久才发现是服务没启动。

**解决方法**：修改 userStore 的 login() 方法，将网络请求放在 try/catch 中，网络错误时抛出「无法连接到 Mock 服务，请检查 JSON Server 是否启动」的错误；账号密码不匹配时抛出「账号或密码错误」。登录页面读取 error.message 动态显示，这样用户能清楚地知道是服务问题还是账号问题。

**问题4：AI 生成的代码中包含 JWT 相关逻辑**

刚开始让 AI 帮忙写登录代码时，它直接生成了一整套 JWT 认证流程，包括 token 生成、token 解码、请求拦截器注入 token、路由守卫检查 token 是否有效等。这些逻辑对于 Day6 来说太复杂了，而且和实训目标不匹配。

**解决方法**：手动删除了所有 JWT 相关代码，保留最简单的账号密码校验逻辑。具体做法是：GET /users 获取全部用户，在前端数组中查找匹配项；匹配成功后直接把用户对象存入 Pinia。这样的代码更短，也更符合实训项目的学习目标。

**问题5：搜索功能不生效**

在二手交易页实现搜索时，起初在 SearchBar 组件中将搜索词 emit 出来，但父组件的过滤逻辑放在了 wrong 位置，导致 keyword 变化时列表没有重新渲染。

**解决方法**：检查 computed 的依赖是否正确，确保 filteredList 的计算属性中使用了 keyword.value。只要 keyword 有变化，computed 就会重新计算并自动更新模板。修复后搜索功能正常。

## 11. 今日反思

今天最大的收获是明白了「用户状态」这个概念其实不只是一个组件内的变量，而是贯穿整个应用的共享信息。通过 Pinia 管理登录状态，我看到了导航栏、发布页、个人中心等页面如何同时响应同一个状态的变化，这让我对「状态管理」有了更具体的认识。

关于状态持久化，我之前总以为这是后端或者框架自动处理的事情，但今天亲手实现了 localStorage 的存取和恢复逻辑，才明白前端状态需要开发人员主动保存和恢复。这种「内存中的状态会丢失」的理解很关键，在真实项目中这个问题会被更复杂的方案解决，但 Day6 的 localStorage 版本让我理解了最底层的原理。

关于功能边界，今天的实训让我意识到，学生项目中实现的「登录/注册」和真实网站的登录不是一回事。真实系统需要后端服务、密码加密、HTTPS、Token、权限校验等一大堆安全措施，而我们只是在前端模拟了一个流程。了解这个边界很重要，不会让我错误地以为当前写的代码可以直接用于生产。

交互优化方面，通过增加 loading、error、empty 状态，我意识到一个好的 Web 应用需要考虑所有可能的状态，而不仅仅是「成功」这一种状态。用户体验的好坏往往就体现在异常状态时的反馈是否及时和友好，而不在于功能是否能用。
