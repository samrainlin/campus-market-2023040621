# 「校园轻集市」—— 前端实训项目答辩资料

> 适用场景：课程答辩 / 实训汇报  
> 技术栈：Vue 3 + Vite + TypeScript + Vue Router + Pinia + Axios + JSON Server + Element Plus  
> 开发周期：7 天

---

## 第一部分 · 项目背景与定位

### 1.1 项目简介

**「校园轻集市」** 是一个面向高校校园生活场景的 PC 端 Web App。核心业务围绕校园日常生活中的四类高频需求展开：

| 业务模块 | 解决的痛点 | 典型场景举例 |
|---------|-----------|-----------|
| 🛒 **二手交易** | 毕业生闲置教材/物品难处理，新生难买到低价二手货 | 线性代数教材九成新 15 元、小米充电宝、宿舍小桌 |
| 🔍 **失物招领** | 校园丢物品后难找到失主，捡到物品后找不到人 | 图书馆丢水杯、操场丢饭卡、宿舍楼下捡雨伞 |
| 🤝 **拼单搭子** | 外卖满减/洗衣液拼单/拼车一个人凑不齐 | 京东洗衣液拼单满 99 减 30、食堂拼餐 |
| 🏃 **跑腿委托** | 快递太多拿不动、排队太长没时间、急件没时间取 | 菜鸟驿站取大箱子、打印店帮排队打印、食堂帮忙带饭 |

### 1.2 项目目标

在 7 天内完成一个**可运行、可演示、可追溯**的前端实训项目，重点锻炼以下能力：

1. 能用 Vue 3 生态完成一个**多页面 Web 应用**的完整开发流程
2. 能使用 **JSON Server** 模拟后端接口，打通前端到数据的完整通路
3. 能用 **Pinia** 管理跨页面共享的前端状态
4. 能使用 **AI Coding 工具辅助开发**，并保持人工判断与代码审查
5. 能形成**工程化意识**——目录划分、组件拆分、类型定义、代码规范、构建检查

### 1.3 技术选型说明

| 技术 | 为什么选它 | 负责的事 |
|------|-----------|---------|
| **Vue 3** | 生态成熟、学习曲线友好、Composition API 代码组织清晰 | 页面组件核心框架 |
| **Vite** | 启动快、HMR 热更新流畅、开箱即用 | 开发服务器 + 生产构建 |
| **TypeScript** | 类型安全，编译时就能发现 bug | 所有页面、组件、接口的类型定义与检查 |
| **Vue Router** | Vue 官方路由，支持命名路由、路由守卫、懒加载 | 多页面导航与动态标题设置 |
| **Pinia** | Vue 官方推荐的状态管理，API 比 Vuex 更简洁 | 用户登录态、收藏数据的跨页面共享 |
| **Axios** | 统一的 Promise HTTP 请求封装 | 调 JSON Server 的 GET / POST 接口 |
| **JSON Server** | 零配置就能得到完整 REST API，适合前端独立开发 | 提供 Mock 后端数据服务（3001 端口） |
| **Element Plus** | 组件丰富，适合快速搭建表单、按钮、提示等交互 | 通用 UI 组件库（可逐步引入） |

### 2.2 核心功能说明

#### 📌 功能 1：列表页数据展示（四个业务模块）

**做了什么**：
- 四个列表页（[TradeView.vue](file:///d:/campus-market/campus-market-seed/src/views/TradeView.vue)、[LostFoundView.vue](file:///d:/campus-market/campus-market-seed/src/views/LostFoundView.vue) 等）的结构一致——页面标题 + 搜索栏 + 卡片列表 + 空状态
- 每个卡片（[ItemCard.vue](file:///d:/campus-market/campus-market-seed/src/components/ItemCard.vue)）展示标题、分类、价格/任务、发布时间、发布人、描述
- 卡片底部添加了「收藏/已收藏」按钮，状态实时更新

**实现要点**：
```ts
// 页面挂载时调接口拿数据
onMounted(async () => {
  loading.value = true
  try {
    dataList.value = await getTrades()
  } catch (e) {
    errorMsg.value = '请检查 Mock 服务是否正常'
  } finally {
    loading.value = false
  }
})
```

**数据结构**（[db.json](file:///d:/campus-market/campus-market-seed/db.json)）：
- `trades`：二手商品（title、price、category、condition、publisher、publishTime、location、image、status）
- `lostFounds`：失物招领（title、type、itemName、location、time、contact、description、status）
- `groupBuys`：拼单搭子（title、type、targetCount、currentCount、deadline、location、publisher、description、status）
- `errands`：跑腿委托（title、taskType、reward、pickupLocation、deliveryLocation、deadline、publisher、description、status）

#### 📌 功能 2：搜索与筛选

**做了什么**：
- 在二手交易页实现了**关键词搜索**功能，可在标题 / 分类 / 地点 / 描述四个字段中模糊匹配
- 使用 computed 响应式计算，输入即过滤，无需点击「搜索」按钮

**关键代码片段**（[SearchBar.vue](file:///d:/campus-market/campus-market-seed/src/components/SearchBar.vue)）：
```vue
<!-- 搜索栏：输入框 + 清空按钮 + 结果统计 -->
<div class="search-bar">
  <input v-model="keyword" placeholder="搜索关键词..." />
  <button v-if="keyword" @click="keyword=''">清空</button>
  <span>匹配到 {{ matched }} 条</span>
</div>
```

**过滤逻辑**：
```ts
const filteredList = computed(() => {
  const kw = keyword.value.trim().toLowerCase()
  if (!kw) return dataList.value
  return dataList.value.filter(item =>
    item.title.toLowerCase().includes(kw) ||
    item.category.toLowerCase().includes(kw) ||
    item.location.toLowerCase().includes(kw) ||
    item.description.toLowerCase().includes(kw)
  )
})
```

#### 📌 功能 3：发布信息流程

**做了什么**：
- 发布页（[PublishView.vue](file:///d:/campus-market/campus-market-seed/src/views/PublishView.vue)）支持 4 类业务切换，切换时自动清空旧字段避免脏数据
- 统一使用 [FormField.vue](file:///d:/campus-market/campus-market-seed/src/components/FormField.vue) 组件做表单项，统一标签 / 必填 / 错误提示样式
- 前端校验：必填项、价格 > 0、拼单人数 ≥ 2、酬劳 ≥ 0
- 提交中按钮置 disabled，防止网络慢时重复提交
- 提交成功后跳转到对应列表页，Network Error 时给出「请检查 Mock 服务」的明确提示
- **登录拦截**：未登录点击发布页会弹提示并跳转到登录页

**提交流程伪代码**：
```
1. 校验表单 → 有错误显示错误文字，不发请求
2. 检查登录状态 → 未登录提示并跳 /login
3. 调 createXxx(data) POST 接口写入 db.json
4. 成功 → 跳对应列表页
5. 失败（如 Mock 服务未启动） → alert 明确错误信息
6. 无论成功失败，finally 把 submitting 恢复为 false
```

#### 📌 功能 4：注册登录 + 状态持久化

**做了什么**：
- **注册页**（[RegisterView.vue](file:///d:/campus-market/campus-market-seed/src/views/RegisterView.vue)）：用户名 ≥ 3 字符、密码 ≥ 6 字符、两次密码一致；通过 `getUsers()` 检查用户名是否重复；通过 `createUser()` POST 写入 db.json；注册成功跳登录页
- **登录页**（[LoginView.vue](file:///d:/campus-market/campus-market-seed/src/views/LoginView.vue)）：前端匹配 username + password，成功后写入 [user store](file:///d:/campus-market/campus-market-seed/src/stores/user.ts) 同时存到 localStorage（key = `campus-market-current-user`）
- **状态恢复**：[App.vue](file:///d:/campus-market/campus-market-seed/src/App.vue) 的 `onMounted` 中调用 `userStore.restoreLogin()`，从 localStorage 读取用户信息并恢复 Pinia 状态
- **退出登录**：导航栏的「退出」按钮调 `userStore.logout()`，清 Pinia + 清 localStorage，并跳回首页

**核心思路**（[user.ts store](file:///d:/campus-market/campus-market-seed/src/stores/user.ts)）：
```ts
// 简化说明（非原代码逐字）
const user = ref<User | null>(null)
const isLoggedIn = computed(() => !!user.value)

function login(username: string, password: string) {
  // 1. 调 getUsers() 拿所有用户
  // 2. 前端遍历匹配 username + password
  // 3. 匹配成功 → 存到 Pinia + localStorage
  // 4. 失败/网络错误 → 抛不同 error message
}

function logout() {
  user.value = null
  localStorage.removeItem('campus-market-current-user')
  router.push('/')
}

function restoreLogin() {
  const saved = localStorage.getItem('campus-market-current-user')
  if (saved) user.value = JSON.parse(saved)
}
```

**导航栏联动**：[AppHeader.vue](file:///d:/campus-market/campus-market-seed/src/components/AppHeader.vue) 根据 `userStore.isLoggedIn` 展示不同内容——未登录显示「登录/注册」按钮，已登录显示「用户名/退出」。

#### 📌 功能 5：收藏功能

**做了什么**：
- [favorite store](file:///d:/campus-market/campus-market-seed/src/stores/favorite.ts) 管理 favorites 数组，每项存 id + type + title + description + location
- 四个 API 方法：`addFavorite`、`removeFavorite`、`toggleFavorite`、`isFavorite`
- 卡片底部按钮显示状态差异——收藏 / 已收藏，颜色和阴影都有视觉区分
- 个人中心（[UserCenterView.vue](file:///d:/campus-market/campus-market-seed/src/views/UserCenterView.vue)）可查看全部收藏并取消收藏

#### 📌 功能 6：个人中心

**做了什么**：
- 展示用户头像、昵称、学院、年级（从 userStore 读取）
- 展示「我的收藏」：从 favoriteStore 读取，点击取消可删除
- 展示「我的发布」：遍历四类业务数据，筛选 publisher = 当前用户名的记录
- 未登录访问时，展示登录引导卡片，引导用户去注册/登录

#### 📌 功能 7：加载 / 错误 / 空状态的用户体验

**做了什么**：
- [LoadingState.vue](file:///d:/campus-market/campus-market-seed/src/components/LoadingState.vue)：页面请求数据时展示加载动画 + 提示文字，避免用户以为页面卡住
- [ErrorState.vue](file:///d:/campus-market/campus-market-seed/src/components/ErrorState.vue)：请求失败时展示错误图标 + 错误消息 + 「重新加载」按钮，用户可手动重试
- [EmptyState.vue](file:///d:/campus-market/campus-market-seed/src/components/EmptyState.vue)：列表无数据时展示「暂无信息」，不让页面一片空白

**为什么重要**：前端应用的用户体验往往体现在异常场景上——数据正在加载、加载失败、加载了但没数据，这些场景都应该给用户明确的反馈，而不是让页面「静默」。

### 3.2 组件化说明

| 组件 | 位置 | 职责 | 复用性 |
|------|------|------|-------|
| **AppLayout** | components/ | 所有页面的统一容器，头部导航 + 内容区 | 10 个页面共用 |
| **AppHeader** | components/ | 顶部导航栏，路由高亮、登录态切换 | 全局共用 |
| **ItemCard** | components/ | 信息卡片，接收 item prop | 四类业务列表页共用 |
| **SearchBar** | components/ | 搜索栏 + 清空按钮 + 统计 | 可复用到所有列表页 |
| **FormField** | components/ | 表单项（label + 必填 * + 错误文字） | 发布页、注册页、登录页共用 |
| **LoadingState** | components/ | 加载状态动画 + 提示文字 | 所有列表页共用 |
| **ErrorState** | components/ | 错误图标 + 错误消息 + 重新加载 | 所有列表页共用 |
| **EmptyState** | components/ | 空状态插画 + 提示文案 | 所有列表页共用 |

**为什么这样拆分**：
- 一个组件「只做一件事」（单一职责原则）
- 样式/逻辑重复的部分都抽到 components/，views/ 只负责「组装页面」
- 以后要加新页面（如 /trade/:id 详情页）时，可以直接复用 ItemCard + LoadingState + ErrorState

### 3.3 数据流向

```
1. 页面挂载 (onMounted)
        ↓
2. 调 API（如 getTrades()）→ Axios → http://localhost:3001/trades
        ↓
3. JSON Server 读 db.json 返回 JSON 数据
        ↓
4. 页面用 ref 保存响应式数据
        ↓
5. <script setup> 中 computed 计算过滤后的列表（搜索 keyword 影响）
        ↓
6. <template> 中 v-for 渲染 ItemCard 列表
        ↓
7. 用户交互（收藏/发布/登录）→ 调用 Store 或 API 更新状态
        ↓
8. 响应式系统自动重新计算并更新视图
```

> 演示总时长建议 **8~12 分钟**，前 6 个演示是核心，第 7 个是加分项

## 第五部分 · AI 协作说明（答辩重点）

### 5.1 使用了什么 AI 工具

**Trae AI（IDE 内置 AI Coding 工具）**，全程用于：

| 使用场景 | 举例 |
|---------|------|
| 代码结构生成 | 生成 PublishView.vue 的模板 + <script setup> 骨架 |
| 组件模板生成 | 生成 ItemCard.vue、FormField.vue、SearchBar.vue 的基础结构 |
| API 方法生成 | 生成 trade.ts、lostFound.ts 等接口模块的基本代码 |
| Pinia store 生成 | 生成 user.ts / favorite.ts 基本框架 |

| 错误排查 | 遇到 `Object is possibly undefined` 时问 AI 可能原因 |
| 文档撰写辅助 | 让 AI 生成 README 的章节框架和检查脚本的骨架 |

### 5.2 AI 生成内容中出现过的问题（答辩时重点讲）

#### 问题 1：过度设计 / 引入未使用的依赖

**场景**：
- 让 AI 生成注册登录页时，它自动写了 **JWT token 生成、解码、axios 请求拦截器注入 token、路由守卫鉴权** 一整套「生产级认证方案」
- 让 AI 生成搜索组件时，它写了一套 **后端接口搜索方案**（调 /search 接口），而项目实际用的是前端 filter 过滤

**影响**：
- 项目复杂度飙升，但这些功能在实训场景下根本用不上
- JSON Server 没有 token 机制，硬塞 JWT 会让登录链路自相矛盾

#### 问题 2：字段名 / 字段值不一致

**场景**：
- AI 在 ErrandeView 里用了 `item.from` / `item.to`，但 db.json 里跑腿委托的字段名是 `pickupLocation` / `deliveryLocation`
- AI 在失物招领的 type 字段里写了 `'lost'` / `'found'`（英文），但 db.json 里实际存的是中文「寻物」「招领」

**影响**：页面虽然不报错，但数据为空字符串，看起来像「功能坏了但不知道为什么」

#### 问题 3：描述不真实 / 不准确

**场景**：
- AI 在生成 README 技术栈时把 **Element Plus** 写进去，但实际上 Day2 阶段我们用的是原生 HTML + CSS，没有在代码里引入 Element Plus
- AI 给收藏功能加了「同步到后端」的逻辑，实际项目只做了前端内存收藏

**影响**：技术栈描述与实际代码不一致，验收时会被质疑代码理解程度

#### 问题 4：缺少用户体验细节

**场景**：
- AI 生成的发布表单提交逻辑**没有提交状态管理**——用户点了提交后按钮一直可用，网络慢时可能重复点击
- AI 生成的列表页**没有空状态**——JSON Server 未启动时页面一片空白，用户不知道发生了什么
- AI 生成的登录页**没有区分错误类型**——账号错、密码错、服务未启动都提示「账号或密码错误」

**影响**：代码看起来「功能可以跑」，但离「用户体验友好」还有距离

### 5.3 自己如何判断和修改 AI 生成内容

**判断原则（每次都问自己这几个问题）**：

| 判断维度 | 具体问题 | 我的处理方式 |
|---------|---------|------------|
| **与真实数据是否一致** | AI 写的字段名和 db.json 一致吗？ | 以 db.json 为权威数据源，不一致就人工改 |
| **是否匹配项目阶段** | AI 建议的复杂度（JWT、路由守卫）符合当前实训要求吗？ | 过度设计就删掉，保留最简可工作版本 |
| **是否真的在项目中使用** | AI 加的库/组件（Element Plus）实际引入了吗？ | 检查 package.json 和实际代码，未使用就删除描述 |
| **异常场景是否覆盖** | API 失败了会怎么样？用户未登录能发发布吗？表单字段空了会怎样？ | 手动补 try/catch、提交状态管理、错误分支提示 |
| **类型是否安全** | AI 写的 any、类型断言是否合理？ | 给每个数据结构写 interface，消除 any |

**具体案例**：
- **案例 1（发布表单）**：AI 给了基础模板 → 我手动补充了 ① submiting 状态管理 ② 类型切换清空旧字段 ③ try/catch 捕获 Mock 服务未启动 ④ 发布人从 userStore 取 ⑤ datetime 格式转换
- **案例 2（登录页）**：AI 给了 JWT 全套方案 → 我全部删除，重写为「前端匹配 username+password + Pinia 存 + localStorage 持久化」的简单方案，并区分网络错误和账号错误的提示语
- **案例 3（favorite store）**：AI 建议把收藏数据 POST 到后端接口 → 我删掉这段逻辑，改为仅在前端 Pinia 内存中维护，因为项目阶段不要求后端持久化
- **案例 4（技术栈描述）**：AI 生成的 README 写了 Element Plus → 我检查实际代码后从技术栈中删除，保证描述与代码一致

### 5.4 以后使用 AI Coding 工具会注意什么

| 注意事项 | 说明 |
|---------|------|
| **先想清楚要什么，再让 AI 生成** | 先自己画好页面结构、定义好数据字段、想清楚数据流，然后给 AI 明确的 Prompt，不要让 AI「自由发挥」 |
| **字段名和类型定义要人工掌控** | 这是前端和数据端的接口约定，一旦不一致就各种 bug。AI 会天马行空命名，必须人工统一 |
| **AI 擅长结构，细节必须人工补** | AI 生成的代码结构通常没问题，但「提交状态管理、类型区分、异常提示、空状态兜底」这些细节往往缺失 |
| **永远不要盲目信任 AI 生成的技术栈描述** | AI 经常「觉得你应该用了什么」就写进描述里，必须人工核对 package.json、main.ts、实际的 import 引用 |
| **保留过程记录（证据卡）** | Day1~Day7 的证据卡是我从 0 到 1 的开发轨迹，比最终成品更能体现学习过程。以后不管做什么项目，都要边做边记录 |
| **区分「能跑的代码」和「好的代码」** | AI 生成的代码往往「能跑」，但距离「易维护、易扩展、异常场景友好」还有距离。好的代码需要人的判断和打磨 |

### 6.2 改进方向建议

**短中期改进（一周内可完成）**：
1. **用户登录持久化扩展**：把 favorite 也写进 localStorage，避免刷新丢失
2. **消息中心动态化**：在 db.json 新增 messages 数组 → 在 userStore 中加未读消息计数 → AppHeader 加红点提示

5. **接入测试**：用 Vitest 写单元测试，Vue Test Utils 写组件测试
6. **响应式适配**：让项目在手机、平板、PC 上都有良好布局
7. **部署上线**：前端部署到 Vercel/Netlify，后端部署到自己的服务器或云平台

## 第七部分 · 总结与反思

### 7.1 对前端项目开发的理解

经过 7 天从零到一完成「校园轻集市」，我对前端项目开发有了更系统的理解：

1. **不是从代码开始，而是从业务和数据开始**  
   最关键的一步其实是 Day3 设计 db.json 的数据结构——一旦字段确定了，API 接口、组件 props、页面渲染逻辑都围绕它展开。后续发现字段名不一致的 bug，根源就是没有在最开始确立「数据源权威」的概念。

2. **页面骨架决定了后续扩展成本**  
   Day2 做的页面骨架和路由看起来是「体力活」，但它决定了后续每个业务页面是按同样模式快速扩展，还是每个页面要重写导航和样式。公共布局 + RouterView 的方式让新增页面成本很低。

3. **状态管理的本质是「跨页面共享数据」**  
   Pinia 不是为了炫技引入的——用户登录状态需要在顶部导航、发布页、个人中心三处同时用到；收藏数据需要在列表页和个人中心两处同步。这些都是单页面内部 reactive 解决不了的问题，需要全局 Store。

4. **异常场景的用户体验往往区分了「作业项目」和「合格产品」**  
   LoadingState、ErrorState、EmptyState、提交中禁用按钮、错误类型区分提示——这些都是 AI 最初生成代码时不会自动帮你加的，但恰恰是用户能感知到的「好体验」。

5. **工程化不是一句口号**  
   TypeScript 的类型检查、目录划分、组件职责分离、`npm run build` 验证、检测脚本检查——这些都是「工程化」的具体体现。项目能通过构建，说明代码没有类型错误；能通过检测脚本，说明目录结构符合预期。

### 7.2 对 AI 协作开发的理解

这是我第一次**系统性地**使用 AI Coding 工具完成一个完整项目，最大的收获不是少写了多少行代码，而是学会了**如何让 AI 为我所用，而不是被 AI 的输出牵着走**。

**我总结的 AI 协作方法论**：

- **先定义清楚要什么**：在让 AI 生成前，我自己先想好字段名、数据结构、页面流程
- **AI 出结构、人出细节**：AI 擅长给我骨架和样板代码，但异常场景、状态管理、业务逻辑细节必须我自己填
- **把 AI 当结对编程伙伴，不当下线**：AI 的每个输出都要人工 review——它会不会引入了未使用的依赖？会不会字段命名不一致？会不会过度设计？
- **证据卡比代码更重要**：过程记录（Day1~Day7 的证据卡）完整记录了「我怎么想的、AI 给了什么、我改了什么、遇到了什么问题、怎么解决的」，这比最终成品更能体现真实掌握程度

**一句话总结**：AI 是一个**非常高效的助手**，但最终判断和审查必须由开发者自己完成。

### 7.3 对工程化过程的理解

**工程化 = 让项目可维护、可扩展、可验证、可交付**

- **可维护**：目录划分清晰、组件职责单一、命名一致。以后别人接手能很快上手
- **可扩展**：以后要加详情页、加搜索、加新业务，只要在现有结构上扩展即可，不需要推翻重写
- **可验证**：TypeScript 让编译阶段就能发现类型错误；检测脚本可以检查文件是否齐全；手动功能走查保证业务链路正常
- **可交付**：`npm run build` 能产出 dist 目录的可部署产物；README + 证据卡 + 本答辩文档构成完整的交付材料

**我对工程化的新理解**：工程化不是最后一天才做的事，而是贯穿每天开发的习惯。从 Day2 的组件命名、Day3 的类型定义、Day4 的表单校验、Day5 的 Store 拆分、Day6 的错误处理，到 Day7 的构建和检测——都是工程化的具体实践。
